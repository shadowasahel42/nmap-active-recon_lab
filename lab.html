Enumeration with Nmap: Full Lab Walkthrough

Reading time: ~24 min Last updated: 6 July 2026 Difficulty: Beginner–Intermediate Tool version: Nmap 7.94

Enumeration is the phase of a security assessment where you go from "there is a network" to "here is what's actually running on it, and here's where it might be weak." Nmap is the tool most people reach for first, not because it's the flashiest option, but because it answers the two most important early questions cheaply: what's alive, and what's listening.

Everything below happened in one sitting, against one lab network, from a single Kali VM. The commands are shown exactly as they were typed, including a couple of mistakes that are worth seeing precisely because they're the kind of thing that happens to everyone the first few times they use this tool.

A note on the target

The primary target in this walkthrough, gravemind.vm (10.6.6.23), is a deliberately vulnerable machine set up for training purposes inside an isolated lab network. It's not a production system and it's not reachable from outside that network.

Objectives

By the end of this walkthrough, you should be able to:

  • Confirm your own machine's network position before scanning anything else.
  • Sweep a subnet to find which hosts are actually up.
  • Explain the practical difference between a SYN scan, a connect scan, and a version scan.
  • Read Nmap's OS-detection output critically rather than taking it as gospel.
  • Use the Nmap Scripting Engine (NSE) to pull specific information out of a service, rather than just knowing a port is open.
  • Recognize what an anonymous FTP login or a misconfigured SMB share actually means for security.

Prerequisites

You should have Nmap installed (this walkthrough uses version 7.94, the version that ships with current Kali releases) and a target you're authorized to scan. If you want to follow along on your own lab, a couple of intentionally vulnerable VMs behind a host-only network adapter is enough — you don't need the exact same setup used here.

Lab topology

The attacking machine is a Kali VM with several interfaces, only one of which matters for this exercise. The target sits on an isolated 10.6.6.0/24 network alongside a handful of other training VMs that aren't touched in this walkthrough.

RoleAddress
Attacker (Kali)10.6.6.1
Target (gravemind.vm)10.6.6.23

Environment verification

Before pointing a scanner at anything, it's worth spending thirty seconds confirming which network you're actually on. It's easy to assume you're talking to the lab network and instead be looking at a Docker bridge or a VPN adapter — and that mistake produces confusing results later that look like Nmap problems but are really addressing problems.

Explanation

The output looks intimidating mostly because of volume, not complexity. The br-internal interface is the one that matters here — it carries the address 10.6.6.1, which matches the lab subnet described in the topology section. The eth0 interface, by contrast, is the VM's connection to the outside internet via NAT (10.0.2.15 is the classic VirtualBox NAT range), and lo is simply the loopback interface every Linux system has.

At this point the output also contained a long run of veth* and br-* interfaces. These can safely be ignored — they belong to Docker's container networking, created automatically because some of the lab's vulnerable-app containers run locally, and they have nothing to do with the actual scan target.

Important observations

  • The address that matters for this lab, 10.6.6.1, is bound to br-internal, not eth0.
  • Every Docker-related interface (docker0 and the veth* pairs) can be mentally filtered out for the rest of this walkthrough.
  • None of these interfaces show a carrier or collision count worth worrying about — the RX/TX error columns are all zero, which is what you want to see before you start troubleshooting connectivity elsewhere.

Common mistakes

Scanning the wrong subnet is the single most common early mistake with multi-adapter Kali VMs. If you run nmap -sn 10.0.2.0/24 instead of 10.6.6.0/24 here, you'll get a confusing "no hosts up" result that has nothing to do with Nmap and everything to do with pointing it at the wrong network.

Confirming the Nmap version

-V

Purpose

Verify which release of Nmap is installed, since flag availability and NSE script behavior can shift between versions.

Syntax

nmap -V

Command & output

kali@kali: ~
┌──(kali㉿Kali)-[~]
└─$ nmap -V
Nmap version 7.94 ( https://nmap.org )
Platform: x86_64-pc-linux-gnu
Compiled with: liblua-5.4.4 openssl-3.0.9 libssh2-1.10.0 libz-1.2.13 libpcre-8.39 libpcap-1.10.4 nmap-libdnet-1.12 ipv6
Compiled without:
Available nsock engines: epoll poll select

Explanation

Nmap 7.94 is current as of this writing and ships with a fairly complete feature set: IPv6 support, Lua for the scripting engine (NSE), and OpenSSL for anything that touches TLS. The "compiled without" line is empty, meaning this build wasn't stripped of any optional features — useful to know later, since a minimal build could silently disable some NSE categories.

Security implications

Version awareness matters on both sides of a scan. An outdated Nmap might lack signatures for newer services, and an outdated target service is exactly what this whole lab is trying to surface.

Host discovery

-sn

With the correct interface confirmed, the next question is simply: which of the 256 possible addresses in 10.6.6.0/24 actually belong to a running machine? A ping sweep answers that without touching a single port.

Purpose

Identify which hosts on the lab subnet are currently up, before spending time port-scanning addresses that aren't even occupied.

Syntax

nmap -sn <target-range>

Command & output

kali@kali: ~
┌──(kali㉿Kali)-[~]
└─$ nmap -sn 10.6.6.0/24
Starting Nmap 7.94 ( https://nmap.org ) at 2026-07-05 21:32 UTC
Nmap scan report for 10.6.6.1
Host is up (0.00085s latency).
Nmap scan report for webgoat.vm (10.6.6.11)
Host is up (0.0012s latency).
Nmap scan report for juice-shop.vm (10.6.6.12)
Host is up (0.00094s latency).
Nmap scan report for dvwa.vm (10.6.6.13)
Host is up (0.00052s latency).
Nmap scan report for mutillidae.vm (10.6.6.14)
Host is up (0.00020s latency).
Nmap scan report for gravemind.vm (10.6.6.23)
Host is up (0.0011s latency).
Nmap scan report for 10.6.6.100
Host is up (0.00059s latency).
Nmap done: 256 IP addresses (7 hosts up) scanned in 2.95 seconds

Explanation

-sn tells Nmap to skip the port-scanning phase entirely and only perform host discovery — in practice a mix of ICMP echo requests, ARP requests (since this is a local subnet), and a couple of TCP probes to common ports, any one of which counts as proof a host is alive. In under three seconds, Nmap checked all 256 addresses in the /24 and reported seven hosts up, including the Kali box itself.

Notice that Nmap resolved reverse-DNS names for most of the hosts (webgoat.vm, dvwa.vm, and so on) but left two addresses as bare IPs. That's simply because those two hosts don't have PTR records configured in the lab's local DNS — it doesn't mean anything is wrong with them.

Important observations

  • Seven hosts responded out of 256 possible addresses — the rest of the subnet is simply unused space reserved for future lab machines.
  • gravemind.vm at 10.6.6.23 is the host this walkthrough focuses on from here forward.
  • A ping sweep this fast (under 3 seconds for a full /24) is only realistic on a local, low-latency network — the same scan against a routed WAN target would take considerably longer and would be far less reliable, since ICMP is commonly filtered.

Common pitfalls

Common pitfall

A ping sweep can under-report live hosts if a target's firewall drops ICMP and doesn't respond on any of the default probe ports. When that happens, the fix isn't to assume the host is down — it's to add -Pn later to skip discovery and scan it directly.

Related commands

-PE -PS -Pn arp-scan

Initial port scan

With one host chosen as the target, the next step is finding out what's actually listening on it. Nmap was run twice here: once with no scan-type flag at all, and once explicitly with -sT, to make the relationship between the two visible.

Default scan

Purpose

Get a fast first look at which of the 1,000 most common TCP ports are open on the target.

Command & output

kali@kali: ~
┌──(kali㉿Kali)-[~]
└─$ nmap 10.6.6.23
Starting Nmap 7.94 ( https://nmap.org ) at 2026-07-05 21:34 UTC
Nmap scan report for gravemind.vm (10.6.6.23)
Host is up (0.0014s latency).
Not shown: 994 closed tcp ports (conn-refused)
PORT    STATE SERVICE
21/tcp  open  ftp
22/tcp  open  ssh
53/tcp  open  domain
80/tcp  open  http
139/tcp open  netbios-ssn
445/tcp open  microsoft-ds

Nmap done: 1 IP address (1 host up) scanned in 0.26 seconds

Explanation

With no scan type specified, Nmap defaults to a TCP SYN scan when it's run with enough privilege, and quietly falls back to a connect scan otherwise. Six ports came back open, which is a fairly rich attack surface for a single host: an FTP server, SSH, DNS, a web server, and two SMB-related ports that will end up being the most interesting part of this entire lab.

"Not shown: 994 closed tcp ports" is Nmap's way of confirming it checked the default 1,000 ports and is compressing the boring result (closed, connection refused) into a single line rather than printing 994 near-identical rows.

Important observations

  • Six services are exposed: FTP (21), SSH (22), DNS (53), HTTP (80), NetBIOS Session Service (139), and Microsoft-DS/SMB (445).
  • The presence of both 139 and 445 together strongly suggests Samba or Windows file sharing — that's confirmed later in the SMB enumeration section.
  • This scan completed in a quarter of a second, which is only possible with raw-socket SYN scanning on a low-latency LAN target.

TCP connect scan

-sT

Purpose

Run the same scan using a full TCP three-way handshake instead of a raw SYN probe, and compare the result.

Syntax

nmap -sT <target>

Command & output

kali@kali: ~
┌──(kali㉿Kali)-[~]
└─$ nmap -sT 10.6.6.23
Starting Nmap 7.94 ( https://nmap.org ) at 2026-07-05 21:38 UTC
Nmap scan report for gravemind.vm (10.6.6.23)
Host is up (0.00049s latency).
Not shown: 994 closed tcp ports (conn-refused)
PORT    STATE SERVICE
21/tcp  open  ftp
22/tcp  open  ssh
53/tcp  open  domain
80/tcp  open  http
139/tcp open  netbios-ssn
445/tcp open  microsoft-ds

Nmap done: 1 IP address (1 host up) scanned in 0.21 seconds

Explanation

The result is identical to the default scan — same six ports, same states. That's expected: -sT completes a full connect() handshake with each port instead of sending a bare SYN packet and analyzing the response, which is slower and noisier on the wire but doesn't require raw-socket privileges. On a clean local network with no filtering in the way, both techniques will usually agree completely.

Security implications

The distinction matters more on a monitored network than it does here. A SYN scan never completes the handshake, so it doesn't always get logged as a full connection by the target application; a connect scan does complete it, and is more likely to show up in application logs, not just firewall or IDS logs.

Related commands

-sS-sA-sW

Explanation

-O requires elevated privileges because it needs to craft and send raw packets with unusual flag combinations, then compare the target's responses against a large database of known OS fingerprints. Here it came back with a range rather than a single version — "Linux 4.15 - 5.8" — which is normal; kernel fingerprinting narrows things down to a plausible band, not an exact build number.

The MAC address prefix (02:42:...) is worth noticing on its own: that's the locally-administered address range Docker assigns to its virtual network interfaces, which lines up with this being a containerized lab VM rather than a physical machine — consistent with what was already suspected from the interface list earlier.

Important observations

  • "Network Distance: 1 hop" confirms the target is directly reachable on the local segment, with no router in between.
  • Nmap explicitly labels this a best-effort estimate ("OS detection performed... please report any incorrect results") rather than a certainty.

Security implications

Knowing the general OS family narrows down which exploits and default configurations are worth checking later, but treating an OS guess as gospel is a common and avoidable mistake — it should be cross-checked against banners and service versions gathered elsewhere in the scan.

Service & version detection

-sV

Knowing that port 21 is open tells you almost nothing on its own — plenty of software can bind to that port. -sV is what turns "something is listening on 21" into "vsftpd 3.0.3 is listening on 21," which is the difference between a guess and a lead.

Common pitfall

-A

Security implications

Anonymous FTP with world-readable files is a classic low-hanging-fruit finding. In a real assessment, the next logical step is simply to download and read those files — which is outside the scope of an enumeration lab focused on Nmap, but is exactly the kind of lead enumeration exists to produce.

SMB enumeration

-p139,445

Ports 139 and 445 flagged earlier as "netbios-ssn" and "microsoft-ds" are the two classic SMB ports. Together they're usually the richest source of information on a Windows or Samba host, because the protocol itself is designed to answer questions about users, shares, and system identity.

The mistyped command

Command

kali@kali: ~
┌──(kali㉿Kali)-[~]
└─$ nmap -A -p139, 445 10.6.6.23
Starting Nmap 7.94 ( https://nmap.org ) at 2026-07-05 21:56 UTC
# the command hung / produced no further output and was cancelled

Explanation

The stray space after the comma in -p139, 445 is the problem. Nmap's port-list parser doesn't tolerate whitespace inside a comma-separated list the way a human eye skims right past it — depending on the shell and Nmap's parsing behavior, this either scans only port 139 or fails to build a valid port list at all, which is why the second port never gets included as intended.

Common mistakes

Common mistake

Whitespace inside -p lists is one of the most frequent typos in Nmap command lines, precisely because it reads fine to a human. The fix is simply to never put a space after the comma: -p139,445, not -p139, 445.

The corrected scan

Purpose

Run full NSE enumeration against both SMB ports together to identify the host's SMB dialect, OS reporting, and security posture.

Syntax

nmap -A -p139,445 <target>

Explanation

This single scan answers several questions at once. The service line confirms this isn't actually Windows file sharing — it's Samba 4.9.5 on Debian impersonating a Windows host, which is exactly what Samba is designed to do. smb-os-discovery picked up the computer name (gravemind), and smb-security-mode reported that message signing is disabled — labelled directly in the output as "dangerous, but default," which is Nmap's own script authors flagging a real weakness rather than the documentation here editorializing.

It's also worth noticing this scan took over seventeen seconds — dramatically longer than anything else in this walkthrough. SMB script enumeration is comparatively heavy: it opens authenticated (or anonymous) sessions and issues several protocol-level queries per script, which costs real time compared to a simple port probe.

Important observations

  • The two ports report slightly different version strings (a broad "3.X - 4.X" range on 139 versus the precise "4.9.5-Debian" on 445) — 445 is the modern SMB-over-TCP port and typically yields the more accurate fingerprint.
  • Message signing is enabled but not required on SMB2, and outright disabled by default on the legacy security-mode check — both point the same direction: sessions can be negotiated without integrity protection.
  • The reported "Windows 6.1" OS string is a Samba compatibility artifact, not a real Windows install — Samba reports a Windows version number to stay compatible with clients that expect one.

Security implications

Security insight

Disabled or optional message signing on SMB opens the door to man-in-the-middle relay attacks, where an attacker positioned on the network can intercept and relay authentication attempts rather than needing to crack a password directly. This is a well-known, well- documented weakness class and one of the most common findings in real internal network assessments.

smb-enum-groups.nse

No "Host script results" block appears here at all, meaning the script ran but returned nothing. Common mistakes

An empty result is easy to misread as "the command failed." It didn't — it completed successfully and connected to the port; it simply had nothing to report. The distinction matters when deciding whether to debug the command or move on.

Security insight

This is the most significant finding in the entire lab. A share literally commented "Confidential Workfiles" grants anonymous read/write access — meaning anyone on the network, without any credentials, can read whatever is stored there and write new files into it. The account_used: <blank> line confirms no authentication was needed to see any of this.

Explanation

Three shares came back. IPC$ is a standard administrative share every SMB server exposes for inter-process communication and is expected to be present; print$ is the equally standard printer-drivers share. Neither is unusual on its own. The third, workfiles, is not a default share — someone configured it deliberately, gave it a comment that flags exactly how sensitive its contents are supposed to be, and left anonymous read/write access enabled on it anyway.

Important observations

  • All three shares report "Anonymous access: READ/WRITE" — none of them require a login of any kind.
  • Write access, not just read, means an attacker could plant a file rather than only read existing ones — a meaningfully worse position than a read-only leak.
  • The share's own comment ("Confidential Workfiles") is effectively self-incriminating documentation of the misconfiguration's severity.

Common pitfalls

It's tempting to stop at "share found" and move on. The comment field and access level are just as important as the share's existence — a share named backups with read-only guest access is a very different finding from one named workfiles with read/write.

Security implications

Anonymous write access to a network share is a direct path to data tampering or, in the right circumstances, to remote code execution — for example if the share's contents are ever executed by another process on the host. This finding alone would justify an immediate remediation recommendation in a real assessment report.

Analysis

Stepping back from the individual commands, the picture that gravemind.vm paints is one host running at least three services worth a closer look: an FTP server with anonymous access enabled, an SSH daemon that was never actually probed in this lab, and a Samba installation with two separate, serious weaknesses — message signing that isn't enforced, and a share that's writable by anyone. None of these individually required exploitation tooling to find; every one of them came from Nmap and its bundled scripts alone.

It's also worth noting what wasn't found. Port 80 (HTTP) and port 53 (DNS) were identified as open early on but never enumerated further in this session — a reminder that an enumeration pass is rarely exhaustive in one sitting, and it's normal for a lab (or a real engagement) to prioritize the most promising leads first and circle back to the rest later.

Lessons learned

  • Confirming your own network position before scanning saves time later — the interface check at the start ruled out an entire category of confusing false negatives.
  • A default scan and an explicit -sT scan can produce identical results, but knowing why they might differ elsewhere is more valuable than the result itself.
  • Timing templates aren't magic — their effect depends entirely on network conditions and scan size, and testing that assumption directly, as this lab did, is more convincing than taking it on faith.
  • Typos like a stray space in a port list or a missing space before an argument are routine, not signs of doing something wrong — the fix is almost always smaller than the confusion it causes.
  • The most serious finding in this entire lab (the writable SMB share) came from a single-purpose NSE script, not from the broader -A flag — knowing the specific scripts by name is worth the extra step of looking them up.
